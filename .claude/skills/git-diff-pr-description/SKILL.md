# Git Diff PR Description Skill

自動執行 `git diff` 取得變更內容，分析後產生一份結構化的 PR / MR 描述。

---

## Step 1：自動取得 Diff

**不要請使用者手動提供 diff**，直接用 bash 依序執行以下指令，取第一個有輸出的結果：

```bash
# 優先取暫存區變更（已 git add 的內容）
git diff --staged

# 若暫存區無內容，改取工作區所有未提交變更
git diff HEAD

# 若以上皆空，取最近一次 commit 的變更
git diff HEAD~1 HEAD
```

**執行邏輯：**
1. 執行 `git diff --staged`，若有輸出 → 使用此結果
2. 若無輸出，執行 `git diff HEAD`，若有輸出 → 使用此結果
3. 若仍無輸出，執行 `git diff HEAD~1 HEAD` → 使用此結果
4. 若三者皆無輸出，告知使用者：目前沒有偵測到任何變更，請確認是否在正確的 git repo 目錄下

同時執行以下指令補充上下文：
```bash
# 取得變更的檔案清單與統計
git diff --staged --stat || git diff HEAD --stat

# 取得目前 branch 名稱（作為 Title 參考）
git branch --show-current
```

---

## Step 2：分析 Diff 內容

取得 diff 後，依序執行：

### 1. 解讀變更範圍
- 找出所有被修改的檔案（`diff --git a/... b/...`）
- 區分：新增檔案、刪除檔案、修改檔案
- 統計大致行數變化（+/- 行）

### 2. 理解變更意圖
從以下線索推斷此次修改的目的：
- 函式 / 變數命名變化
- 新增或移除的邏輯分支
- import / dependency 變動
- 設定檔、環境變數異動
- 測試案例的新增或修改
- 註解與文件更新

### 3. 評估風險
考量以下面向：
- 是否涉及資料庫 schema 或 migration？
- 是否更動 API 介面（endpoint、參數、回傳格式）？
- 是否有向下相容性問題？
- 是否刪除或重構了核心邏輯？
- 是否缺少對應的測試覆蓋？

---

## Step 3：輸出格式

產生以下四個區塊，使用繁體中文（除非使用者指定其他語言）：

```
## Title
（一句話，動詞開頭，說明「做了什麼」，50 字以內）
例：新增使用者登入 JWT 驗證機制

---

## Description
（條列式，說明此次變更的目的與內容）

- 背景：為什麼需要這個改動？
- 變更內容：
  - 異動檔案 1：做了什麼
  - 異動檔案 2：做了什麼
  - ...
- 預期效果：這個改動會帶來什麼改善？

---

## How to Verify
（具體的操作步驟，讓 reviewer 能自行驗證功能正確）

1. 步驟一
2. 步驟二
3. 預期結果：應看到 / 應發生 XXX

---

## Risk
（潛在風險，若無風險請說明「低風險」並說明原因）

- 🔴 高風險：...（若有）
- 🟡 中風險：...（若有）
- 🟢 低風險：...（若僅有此類）
- 建議：若有緩解方式請一併說明
```

---

## 品質要求

- **Title** 必須一句話說清楚，避免「修改」、「更新」等模糊動詞，優先用「新增」、「修正」、「重構」、「移除」、「優化」
- **Description** 要能讓不熟悉此功能的 reviewer 看懂
- **How to Verify** 必須可操作，避免「測試看看」這類空泛描述；若是純重構或文件變更，說明如何確認行為不變
- **Risk** 要誠實評估，不可一律填「低風險」；若涉及 migration、API 變更、核心邏輯修改，必須標示對應風險等級

---

## 邊界情況

| 情況 | 處理方式 |
|------|----------|
| diff 過大（>500 行） | 先摘要主要變更檔案，再逐區塊分析 |
| 純格式調整（whitespace / lint） | Title 直接說明，Risk 標為低風險 |
| 只有測試檔變動 | Description 說明測試涵蓋範圍 |
| 多個不相關的變更混在一起 | 建議使用者拆分 PR，但仍產出完整描述 |
| 看不出意圖的變更 | 在 Description 標註「需確認」並提示使用者補充說明 |
